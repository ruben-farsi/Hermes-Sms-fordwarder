Info / Mejoras

Arquitectura limpia (Domain/Application/Infrastructure/Presentation) con plugin nativo Android para SMS. Se encontraron 2 hallazgos criticos relacionados con el plugin nativo (botToken en SharedPreferences sin cifrado), 3 de severidad alta (ReDoS potencial, sin SSL pinning, BroadcastReceiver exportado), y varios de severidad media/baja.

Hallazgos Criticos

Critica Bot Token almacenado en texto plano en SharedPreferences

Archivo: plugins/withSmsListener.js:342-348 (SmsListenerModule.kt)

Descripcion: El token del bot de Telegram se guarda directamente en SharedPreferences con la clave "telegram_token" sin ningun tipo de cifrado. En dispositivos rooteados o con acceso fisico, este token puede ser extraido facilmente, permitiendo al atacante enviar mensajes en nombre del bot.

Evidencia:

.putString("telegram_token", token)
.putString("telegram_chat_id", chatId)

Recomendacion: Migrar el almacenamiento del botToken a expo-secure-store (Keychain/Keystore) en lugar de SharedPreferences. El chatId puede permanecer en AsyncStorage/SharedPrefs pero el token DEBE estar en almacenamiento seguro con cifrado hardware-backed.

OWASP: A02:2021 - Cryptographic Failures / A07:2021 - Identification and Auth Failures

Critica ID generado con Date.now() + Math.random() — previsible

Archivos: src/application/GestionarReglas.ts:34, src/application/GestionarAutoRespuestas.ts:7, src/application/EvaluarYReenviarSms.ts:132

Descripcion: Los IDs de entidades (reglas, mensajes, pendientes) se generan con Date.now().toString(36) + Math.random().toString(36).substring(2). Este metodo es predecible: un atacante que conozca el tiempo aproximado de creacion puede reproducir el ID. Además, Math.random() no es criptograficamente seguro.

Impacto: Colision de IDs, potencial sobrescritura de registros en escenarios de alta concurrencia.

Recomendacion: Usar crypto.randomUUID() (disponible en React Native 0.73+) o una libreria como uuid con crypto.getRandomValues().

Hallazgos de Severidad Alta

Alta ReDoS potencial en evaluacion de regex de reglas

Archivo: src/domain/services/EvaluadorDeReglas.ts:58-62

Descripcion: El usuario define patrones regex que se ejecutan directamente con new RegExp(patron, 'i') sobre cada SMS recibido. Un patron malicioso (ej: (a+)+$) puede causar catastrofica backtracking, bloqueando el hilo principal o el servicio nativo.

Impacto: Denial of Service — el servicio de reenvio se congela procesando un solo SMS.

Recomendacion: (a) Agregar un timeout para la ejecucion de regex, (b) validar que el patron no exceda una complejidad maxima (ej: max 50 chars, sin grupos anidados), (c) ejecutar regex en un Worker/isolated thread con timeout.

OWASP: A05:2021 - Security Misconfiguration

Alta Sin SSL Pinning en peticiones HTTP nativas (Android)

Archivo: plugins/withSmsListener.js:216-230 (SmsForwarderService.kt)

Descripcion: El HttpURLConnection del servicio nativo confia en la cadena de certificados del sistema sin implementar SSL pinning. Un atacante con control de red (MITM) o un certificado CA comprometido puede interceptar el botToken y los SMS reenviados.

Impacto: Exfiltracion de botToken y contenido de SMS via MITM.

Recomendacion: Implementar certificate pinning con OkHttp + CertificatePinner para el dominio api.telegram.org. Alternativamente, usar la libreria react-native-ssl-pinning.

OWASP: A02:2021 - Cryptographic Failures / A05:2021 - Security Misconfiguration

Alta BroadcastReceiver SmsReceiver exportado sin permisos

Archivo: plugins/withSmsListener.js:425-438

Descripcion: El SmsReceiver se declara con android:exported="true", lo que permite que cualquier app en el dispositivo envie intents falsos simulando SMS recibidos. Esto podria usarse para inyectar SMS falsos y disparar reenvios no autorizados.

Evidencia:

'android:exported': 'true',
'action': 'android.provider.Telephony.SMS_RECEIVED'

Impacto: Un atacante local puede forzar reenvios de SMS falsos a Telegram.

Recomendacion: (a) Agregar android:permission="android.permission.BROADCAST_SMS" al receiver, (b) validar en onReceive() que el intent action sea exactamente SMS_RECEIVED_ACTION (ya esta), (c) considerar exported="false" si solo el sistema lo invoca (requiere API check).

Hallazgos de Severidad Media

Media Sin validacion de esquema en respuestas JSON.parse

Archivos: src/infrastructure/persistence/RepositorioMensajesAsyncStorage.ts:20, src/infrastructure/persistence/RepositorioAutoRespuestasAsyncStorage.ts:17

Descripcion: Se realizan JSON.parse() sin try/catch sobre datos leidos de AsyncStorage. Si el dato esta corrompido o fue manipulado, la app crashea sin manejo de error.

Recomendacion: Envolver todo JSON.parse() en try/catch con fallback a valor por defecto.

OWASP: A04:2021 - Insecure Design

Media Webhook URL sin validacion de rango privado amplia

Archivo: src/infrastructure/webhook/EnviadorWebhookHttp.ts:10-14

Descripcion: La lista de hosts bloqueados es limitada. No cubre rangos IP privados como 10.x.x.x, 172.16.x.x, 192.168.x.x, ni dominios que resuelven a IPs internas (DNS rebinding). Un webhook malicioso podria exfiltrar SMS a un servidor interno.

Recomendacion: Usar una libreria como net o ip para validar que la IP resolveda a un rango publico. Bloquear todos los rangos RFC 1918 y link-local.

OWASP: A01:2021 - Broken Access Control / A10:2021 - SSRF

Media Sin rate limiting en envio de mensajes

Archivos: src/infrastructure/telegram/EnviadorTelegramApi.ts, src/application/ProcesarColaPendientes.ts

Descripcion: No hay limitacion de tasa (rate limit) en los envios a Telegram. Un pico de SMS (ej: spam) puede agotar la cuota de la API de Telegram (30 msg/sec) y causar errores 429, o generar costos excesivos si se usa un servicio pagado.

Recomendacion: Implementar un token bucket o cola de envio con delay entre peticiones (min 33ms entre envios).

OWASP: A04:2021 - Insecure Design

Media WakeLock sin timeout de liberacion garantizada

Archivo: plugins/withSmsListener.js:103-107

Descripcion: El WakeLock se adquiere con timeout de 10 minutos (acquire(10*60*1000L)), pero si el servicio es matado por el sistema antes de ACTION_STOP, el wakelock puede quedar retenido. Además, no hay verificacion de isHeld() antes de liberar.

Recomendacion: Agregar onDestroy() o onTaskRemoved() con liberacion defensiva del wakelock.

OWASP: A06:2021 - Vulnerable Components

Media Sin proteccion contra intent spoofing en SmsReceiver

Archivo: plugins/withSmsListener.js:33-55 (SmsReceiver.kt)

Descripcion: Aunque se valida intent.action == SMS_RECEIVED_ACTION, en Android 13+ apps maliciosas pueden enviar intents con la misma action. El sistema garantiza que solo el sistema envia SMS_RECEIVED, pero en dispositivos con custom ROMs o Android modificado, esta garantia no existe.

Recomendacion: Validar intent.extras contenga "pdus" (solo el sistema lo agrega). Agregar logging de la fuente del intent.

OWASP: A01:2021 - Broken Access Control

Hallazgos de Severidad Baja

Baja console.error en produccion (HeadlessTask)

Archivo: SmsHeadlessTask.ts:15

Descripcion: Se usa console.error en el headless task. En produccion, estos logs pueden ser capturados por otras apps en Android (logcat) exponiendo informacion sensible (errores con tokens o contenido de SMS).

Recomendacion: Remover console.error en build de produccion con __DEV__ flag o usar un logger que se deshabilite en release.

OWASP: A09:2021 - Security Logging Failures

Baja Sin validacion de longitud maxima en cuerpo de SMS

Archivo: src/application/EvaluarYReenviarSms.ts

Descripcion: No hay limite en el tamaño del cuerpo del SMS procesado. Un SMS extremadamente largo (concatenated SMS de 100+ segmentos) podria causar problemas de memoria o exceder el limite de Telegram (4096 chars por mensaje).

Recomendacion: Truncar o rechazar mensajes mayores a 4096 chars (limite Telegram API).
Baja Sin proteccion de pantalla (screenshot prevention)

Descripcion: La app muestra tokens de Telegram y contenido de SMS sin prevenir capturas de pantalla. En dispositivos sin screen security flag, un atacante con acceso fisico o malware puede capturar informacion sensible.

Recomendacion: Agregar android:secure="true" en las activities sensibles o usar react-native-screenshield.

OWASP: A02:2021 - Cryptographic Failures

Baja Contenedor singleton sin limpieza de suscripciones

Archivo: src/infrastructure/container/ContenedorDeDependencias.ts

Descripcion: El singleton mantiene el listener de red activo indefinidamente. No hay metodo dispose() para cancelar suscripciones. En testing o hot-reload, esto causa memory leaks.

Recomendacion: Implementar interface Disposable con metodo dispose() que llame cancelarMonitoreoRed.

OWASP: A04:2021 - Insecure Design

Observaciones y Mejoras Recomendadas

Info Arquitectura DDD bien implementada

La separacion en capas (Domain / Application / Infrastructure / Presentation) es correcta. Los puertos (interfaces) estan bien definidos y la inversion de dependencias via contenedor singleton funciona adecuadamente. +

Info Idempotencia implementada correctamente

El mecanismo de duplicados con ventana de 10s en EvaluarYReenviarSms es una buena practica. +

Info Timeout en llamadas de red (10s)

Tanto Telegram como Webhook usan AbortController con timeout de 10s. Buena practica. +

Info Webhook failure no bloquea flujo principal

El envio de webhook se ejecuta en try/catch silencioso — correcto, un fallo en webhook no debe impedir el reenvio a Telegram. +

Info Cobertura de tests

Hay tests unitarios para 7 archivos (application + domain + config). Faltan tests para: infraestructura (persistent, network, sms, telegram, webhook), presentacion (hooks, screens), y el plugin nativo.

Recomendacion: Agregar tests de integracion para el contenedor y mocks de NativeModules.

Info Sin obfuscation de c

Descripcion: El codigo JS no esta ofuscado en build de produccion. Un atacante con acceso al APK puede extraer facilmente el bundle y analizar la logica.

Recomendacion: Configurar metro.config.js con minification y considerar herramientas como javascript-obfuscator.

Info Sin ofuscacion de c

El codigo JS no esta ofuscado en build de produccion. Un atacante con acceso al APK puede extraer el bundle y analizar la logica de negocio, incluyendo validaciones y endpoints.

Recomendacion: Configurar metro.config.js con minification y considerar javascript-obfuscator para builds de release.

Info Contenedor singleton sin limpieza de suscripciones

El singleton mantiene el listener de red activo indefinidamente. No hay metodo dispose() para cancelar suscripciones. En testing o hot-reload, esto causa memory leaks.

Recomendacion: Implementar interface Disposable con metodo dispose() que llame cancelarMonitoreoRed.
