Hallazgos de Seguridad

SCR-001 A02-Crypto Failures Crítico
Bot Token almacenado en texto plano en SharedPreferences (plugin nativo)
📁 plugins/withSmsListener.js:342-348 — SmsListenerModule.kt generado
prefs.edit() .putString("telegram_token", token) .putString("telegram_chat_id", chatId) .apply()
Impacto: En dispositivos rooteados o con acceso ADB, el token del bot de Telegram puede ser extraído de SharedPreferences. Esto permite al atacante enviar mensajes en nombre del bot y acceder al chat del usuario.
Fix propuesto: 
// Migrar a EncryptedSharedPreferences o Android Keystore
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()
val prefs = EncryptedSharedPreferences.create(context, "sms_secure", masterKey, ...)
SCR-002 A02-Crypto Failures Crítico
IDs generados con Date.now() + Math.random() — predecibles y colisionables
📁 src/application/GestionarReglas.ts:34, GestionarAutoRespuestas.ts:7, EvaluarYReenviarSms.ts:132
private generarId(): string { return Date.now().toString(36) + Math.random().toString(36).substring(2); }
Impacto: Math.random() no es criptográficamente seguro. En alta concurrencia, colisiones de ID pueden causar sobrescritura de registros (mensajes, reglas).
Fix propuesto: 
import { randomUUID } from 'crypto'; // RN 0.73+
private generarId(): string {
    return randomUUID();
}
SCR-003 A05-Misconfiguration Alto
ReDoS potencial — regex de usuario sin timeout ni validación de complejidad
📁 src/domain/services/EvaluadorDeReglas.ts:58-62
private coincidePorRegex(texto: string, patron: string): boolean { try { const regex = new RegExp(patron, 'i'); return regex.test(texto); } catch { return false; } }
Impacto: Un patrón como (a+)+$ causa backtracking exponencial. El hilo principal o el ForegroundService se congela procesando un solo SMS.
Fix propuesto: 
// Ejecutar regex con timeout usando Worker
const TIMEOUT_MS = 100;
const promise = Promise.race([
  regexTestPromise(texto, patron),
  timeout(TIMEOUT_MS)
]);
SCR-004 A02-Crypto Failures Alto
Sin SSL Pinning en HttpURLConnection nativo (SmsForwarderService.kt)
📁 plugins/withSmsListener.js:216-230
val url = URL("https://api.telegram.org/bot" + token + "/sendMessage") val conn = url.openConnection() as HttpURLConnection conn.requestMethod = "POST"
Impacto: MITM puede interceptar el botToken y el contenido de los SMS. La confianza en el CA system store es insuficiente para apps que manejan datos sensibles.
Fix propuesto: 
// Agregar OkHttp CertificatePinner
val client = OkHttpClient.Builder()
    .certificatePinner(CertificatePinner.Builder()
        .add("api.telegram.org", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
        .build())
    .build()
SCR-005 A01-Broken Access Control Alto
BroadcastReceiver SmsReceiver exportado=true sin permiso de protección
📁 plugins/withSmsListener.js:425-438
'android:exported': 'true', 'action': 'android.provider.Telephony.SMS_RECEIVED'
Impacto: Cualquier app en el dispositivo puede enviar intents falsos al receiver, inyectando SMS falsos que disparan reenvíos a Telegram.
Fix propuesto: 
// Agregar permiso de protección del sistema
'android:permission': 'android.permission.BROADCAST_SMS',
// O cambiar a exported="false" (solo sistema invoca)
SCR-006 A10-SSRF Medio
Webhook URL con validación insuficiente de rangos IP privados
📁 src/infrastructure/webhook/EnviadorWebhookHttp.ts:10-14
const HOSTS_BLOQUEADOS = [ '169.254.169.254', '127.0.0.1', 'localhost', '0.0.0.0', '[::1]', ];
Impacto: No bloquea 10.x.x.x, 172.16.x.x, 192.168.x.x. Un usuario puede configurar un webhook a un servidor interno y exfiltrar SMS.
Fix propuesto: 
import ip from 'ip';
function isPrivate(host: string): boolean {
  try { return ip.isPrivate(host); } catch { return true; }
}
SCR-007 A04-Insecure Design Medio
JSON.parse sin try/catch en repositorios de persistencia
📁 RepositorioMensajesAsyncStorage.ts:20, RepositorioAutoRespuestasAsyncStorage.ts:17
const registros = JSON.parse(datos) as Array<Record<string, unknown>>;
Impacto: Si AsyncStorage está corrompido, la app crashea al iniciar sin posibilidad de recuperación.
Fix propuesto: 
try { return JSON.parse(datos); }
catch { await AsyncStorage.removeItem(CLAVE); return []; }
SCR-008 A04-Insecure Design Medio
Sin rate limiting en envío a Telegram
📁 src/infrastructure/telegram/EnviadorTelegramApi.ts
Impacto: Pico de SMS puede agotar cuota (30 msg/sec) causando 429s. Sin backoff, reintentos inmediatos amplifican el problema.
Fix propuesto: 
// Token bucket: 1 mensaje cada 33ms mínimo
const MIN_INTERVAL = 33;
async enviarMensaje(...) {
  await this.rateLimiter.acquire();
  // ... fetch
}
SCR-009 A04-Insecure Design Medio
WakeLock sin liberación defensiva en onDestroy
📁 plugins/withSmsListener.js:103-107
wakeLock?.acquire(10*60*1000L)
Impacto: Si el servicio es matado por Doze Mode sin recibir ACTION_STOP, el WakeLock queda retenido drenando batería.
Fix propuesto: 
override fun onDestroy() {
    if (wakeLock?.isHeld == true) wakeLock?.release()
    super.onDestroy()
}
